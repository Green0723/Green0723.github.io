PHP反序列化–字符串逃逸

1.为什么逃逸字符串是s:5:”photo”;s:10:”config.php”;}？
  1. 提前闭合nickname字段：用 ";} 结束当前字段
  2. s:5:"photo"; - 定义字段名为”photo”
  3. s:10:"config.php"; - 定义字段值为”config.php”
  4. 合起来：重新定义整个photo字段，将其值设置为”config.php”

通过序列化注入，把photo字段的值从图片路径改成config.php文件路径，从而读取敏感文件。

2.为什么选择要”替换”photo字段的值？

对比其他字段，只有只有 photo 字段会触发 file_get_contents() 函数调用，会读取 $profile['photo'] 指向的文件并显示！！

在profile.php中可以看出photo字段可以进行base64编码，然后被打印出来，所以，这个显示就用来显示flag。

逃逸段需覆盖敏感字段（如 photo），而非无关字段（如 email）。 photo 字段通常用于文件路径或资源加载，覆盖它可能触发文件包含漏洞（如 config.php 含敏感代码）。 email 字段一般为普通数据，篡改后危害性较低。

  1. 它是唯一会触发文件读取的字段
  2. 通过控制它的值可以实现任意文件读取
  3. 最终目标是读取config.php中的flag

若代码中存在以下特征，说明未校验数据完整性，存在安全风险：

  • 1.直接反序列化用户输入

    风险：攻击者可提交任意序列化字符串（如包含恶意类的对象），触发反序列化漏洞

  • 2.缺少签名校验

    无哈希签名：序列化后的数据未附加 HMAC 签名，或签名未与服务端密钥验证

    若未校验签名，攻击者可篡改数据后重新序列化，绕过检测

  • 3.未限制允许的类

    风险：攻击者可构造包含 __destruct() 或 __wakeup() 魔术方法的恶意类对象，触发任意代码执行。

    SQL注入

    如何判断是否存在 SQL 注入？

    SQL 注入的常见特征和验证方法：

    • 输入可控性：用户输入直接拼接到 SQL 语句中（如 $`sql = "SELECT ... WHERE id='`$_GET[id]'"）。
    • 未过滤特殊字符：未对 ‘、”、\、#、– 等符号进行转义或过滤。
    • 错误回显：输入异常字符（如 ‘）时，页面返回数据库错误（如 MySQL Syntax Error）。
    • 布尔盲注验证：构造 1’ AND 1=1# 和 1’ AND 1=2#，观察页面响应差异。
    • 时间盲注验证：构造 1’ AND SLEEP(5)#，观察页面响应时间。

    SQL 注入的常见表达

    • 闭合引号：' OR 1=1#、" UNION SELECT 1,2,3#
    • 联合查询：UNION SELECT 1,version(),database()
    • 报错注入：AND extractvalue(1,concat(0x7e,(SELECT @@version)))
    • 布尔盲注：AND (SELECT SUBSTR(password,1,1)='a')
    • 时间盲注：AND IF(1=1,SLEEP(5),0)

文件上传漏洞的常见特征

  • 1.完全没有校验

    代码直接接收文件并保存，不做任何检查。

    代码特征：

    • 代码中看不到任何与文件类型、扩展名、内容相关的检查函数。
    • 直接从前端表单获取文件名，并保存到服务器目录。

    风险： 攻击者可以上传 .php, .jsp 等恶意脚本，直接获取服务器权限。

  • 2.仅在前端进行校验

    代码特征：

    • 服务器端代码非常简单，与“完全没有校验”的例子类似。
    • 在前端HTML或JavaScript中有文件类型检查的逻辑。

    风险： 攻击者使用 Burp Suite 等工具拦截请求，修改文件名或内容类型（Content-Type）即可轻松绕过。略微常见

  • 3.黑名单策略，且名单不完整

    服务器端有一个“不允许”的扩展名列表（黑名单），但名单可能不全或被绕过。

    代码特征：

    • 有一个数组或列表包含如 ['php', 'exe', 'jsp'] 等危险扩展名。
    • 检查用户文件名是否在这个列表中。

    绕过方法：双写，大小写混写绕过，更改后缀为php3,php4,php5,phtml,pht,php7,inc,phps,上传配置文件

  • 4.校验文件类型（MIME Type）

    检查 HTTP 请求头中的 Content-Type。

    代码特征：

    • 代码检查 $_FILES['file']['type'] 或 Part.getContentType() 等。

    风险： Content-Type 是由浏览器或攻击工具直接发送的，完全可以被伪造。一个文本文件可以被设置为 image/jpeg。

  • 5.检查文件内容头

    这是比检查MIME Type更可靠的方法，但并非绝对安全。

    代码特征：

    • 代码读取文件的前几个字节（文件头）来判断实际类型

    即使文件内容安全，但文件名和路径处理不当也会引发问题。

    代码特征：

    绕过：Content-Disposition: form-data; name="file"; filename="<script>alert(document.cookie)</script>.jpg"

安全的上传代码应具备的特征：

  1. 白名单校验： 只允许明确安全的扩展名（如 .jpg, .png, .pdf）。
  2. 类型与内容双重校验： 同时检查MIME Type（虽然可伪造，但增加难度）和文件头（Magic Number）。
  3. 重命名文件： 使用随机生成的文件名（如UUID）保存，避免文件名冲突和攻击。
  4. 隔离存储： 将上传的文件存储在Web服务器根目录之外，通过脚本（如PHP）代理访问。如果必须放在Web目录，应使用专门的、禁用了脚本执行权限的目录。
  5. 限制文件大小： 防止拒绝服务攻击。
  6. 使用可靠的工具库： 如Apache Commons FileUpload的 getFileName() 方法可以避免一些路径遍历问题。

总结：如何在代码审计中识别？

当你审查文件上传功能的代码时，问自己以下几个问题：

  1. 信任来源？ 代码是否直接信任了来自客户端的任何数据（如文件名、Content-Type）？
  2. 校验了什么？ 它校验了扩展名、MIME Type还是文件内容？
  3. 策略是什么？ 是黑名单（不安全）还是白名单（推荐）？
  4. 最终存储是什么？
    • 存储的文件名是随机的吗？还是用户原来的文件名？
    • 文件最终存储在Web目录下吗？如果不是，风险大大降低。
    • 如果必须存储在Web目录，是否有 .htaccess（Apache）或服务器配置禁止该目录执行脚本？
